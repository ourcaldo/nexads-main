## Critical Issues (5)

### C-1 -- Manager() leaks a background process; Lock is dead code

**Location:** `automation.py:67-68`

`multiprocessing.Manager()` spawns a persistent background server process that is never shut down. The `self.lock` it creates is also never used anywhere -- workers run in separate processes with no shared state. Both lines are dead weight that waste resources.

---

### C-2 -- spawn passes copies, not shared state; CTR is broken with >1 thread

**Location:** `automation.py:144-149`

`ads_session_flags` and `pending_ads_sessions` are passed to `multiprocessing.Process` with spawn -- they are copied, not shared. Each worker gets its own independent copy. With 3 threads, `pending_ads_sessions` gets decremented 3 separate times from its original value, meaning 3x the intended ad sessions fire. The parent process also never sees any mutations, so summary stats are always wrong.

---

### C-3 -- Double-counting: both branches fire simultaneously

**Location:** `worker.py:83-87`

```python
if ctx.pending_ads_sessions > 0:
    is_ads_session = True
    ctx.pending_ads_sessions -= 1
elif session_count < len(ctx.ads_session_flags):
    is_ads_session = ctx.ads_session_flags[session_count]
```

When `pending_ads_sessions > 0`, the code sets `is_ads_session = True` AND decrements the counter, but then the `elif` also reads from `ads_session_flags` independently on the next session. Both systems fire uncoordinated, resulting in more ad sessions than the configured CTR.

---

### C-4 -- successful_ads_sessions incremented every activity loop iteration

**Location:** `worker.py:203-208`

```python
while remaining_time > 0:
    activity_success = await _perform_activity(...)
    if is_ads_session and activity_success:
        successful_ads_sessions += 1  # fires on EVERY loop tick
```

A 60-second stay with 3-second activity intervals = 20 increments per session instead of 1.

---

### C-5 -- successful_ads_sessions incremented a second time after interact_with_ads

**Location:** `worker.py:214-226`

After the activity loop already over-counted (C-4), it gets incremented again if `interact_with_ads` succeeds. The success counter is meaningless.

---

## High Issues (7)

### H-1 -- Summary stats always print zero

**Location:** `automation.py:160-163`

`self.session_counts` and `self.successful_sessions` are dicts on the parent process. Workers run in separate processes and never write back to them. The final summary always prints 0/0.

---

### H-2 -- TypeError if stop() called before start()

**Location:** `automation.py:189`

```python
runtime = datetime.now() - self.start_time  # start_time is None -> TypeError
```

Calling `stop()` before `start()` (e.g. on import error) crashes instead of exiting cleanly.

---

### H-3 -- Proxy parsing crashes on @ in password or IPv6 addresses

**Location:** `setup.py:59-60,69`

```python
auth, server = proxy.split('@')   # crashes if password contains @
host, port   = server.split(':')   # crashes on IPv6 like [::1]:8080
```

Any proxy with a special character in the password silently raises `ValueError` and returns `None` (no proxy used, no warning).

---

### H-4 -- Wrong selector variable used for cookie button

**Location:** `referrer.py:111-112`

The code loops over multiple candidate selectors to find a cookie button, but uses `selector` (the last value from the loop) instead of the variable holding the matched button. This means the wrong button may be clicked or an error raised.

---

### H-5 -- UnboundLocalError on context in cleanup

**Location:** `worker.py:107,237-245`

```python
context = await browser.new_context()  # line 107
# if this raises, 'context' is never assigned
...
finally:
    await process_ads_tabs(context, ...)  # NameError/UnboundLocalError
```

---

### H-6 -- Column stretch ratios are inverted vs. their comments

**Location:** `config_window.py`

The URL table column widths have their stretch comments reversed -- the "URL" column is given a narrow stretch while the narrow toggle columns are given wide stretches.

---

### H-7 -- CTR drops to 0% after 100 sessions when max_time=0

**Location:** `automation.py:98-105`

When `session.max_time = 0`, `total_sessions` is hardcoded to 100. After 100 sessions, `ads_session_flags` is exhausted and no more ad sessions ever fire.

---

## Medium Issues (9)

### M-1 -- Stale current_domain after tab switch

**Location:** `activities.py`

`current_domain` is captured once at the start of `perform_random_activity`. After a tab switch or navigation, it's stale -- random clicks may filter against the wrong domain.

---

### M-2 -- replace('www.', '') mangles domains mid-string

**Location:** `referrer.py:152`

```python
"www.example.com".replace('www.', '')   # -> "example.com" (correct)
"powww.example.com".replace('www.', '') # -> "poexample.com" (wrong)
```

Should use `removeprefix('www.')` or a regex anchored to start.

---

### M-3 -- max_time unit inconsistency

**Location:** `worker.py:116-117`

`max_time` is in minutes in the config but compared against `time.time()` which returns seconds. The `* 60` conversion is present, but `stay_time = random.randint(url_data['min_time'], url_data['max_time'])` on line 194 is in seconds with no annotation. Maintenance hazard mixing units without comments.

---

### M-4 -- Mobile screen cap is larger than desktop (inverted)

**Location:** `setup.py:43-44`

```python
screen = Screen(max_width=2800, max_height=2080) if device_type == 'mobile' \
        else Screen(max_width=7680, max_height=4320)
```

Mobile gets capped at 2800x2080, desktop at 7680x4320. These are backwards -- mobile screens are smaller than desktop monitors.

---

### M-5 -- Ad tabs on the source domain are silently skipped

**Location:** `tabs.py:126-128`

`process_ads_tabs` skips any tab whose URL matches a configured domain. If an ad happens to be on the same domain (self-promo ads), it's never processed.

---

### M-6 -- Request interceptor never torn down

**Location:** `referrer.py:15-21`

`setup_request_interceptor` adds a route handler that blocks all Google Analytics/Tag Manager requests. It's never removed, so it persists for the entire session -- including on pages where it may block legitimate functionality needed for organic search simulation.

---

### M-7 -- Lambda recreated every URL iteration

**Location:** `worker.py:174-177`

The `random_navigation` lambda captures `_ensure_tab`, `_smart_click`, etc. and is recreated on every iteration of the URL loop. Minor but unnecessary allocation in a hot path.

---

### M-8 -- Bare except silently discards all config errors

**Location:** `config_window.py:186-187`

If `config.json` is malformed or has unexpected structure, the GUI silently falls back to defaults and overwrites the file on save -- destroying the user's config without warning.

---

### M-9 -- Session estimation fires incorrectly

**Location:** `automation.py:98`

`if not self.config['session']['enabled'] or self.config['session']['count'] == 0` -- the estimation branch fires when `enabled=True, count=0`, computing a `total_sessions` based on `max_time`. But if `enabled=True`, the intent is to use `session.count`. Leads to wrong CTR calculation.

---

## Enhancements (5)

### E-1 -- Browser startup uses global delay; needs own setting

**Location:** `setup.py:76-78`

Browser startup uses the global delay config -- should have its own dedicated `startup_delay` setting.

---

### E-2 -- press(char) breaks on non-ASCII keywords

**Location:** `referrer.py:175-177`

`press(char)` breaks on non-ASCII keywords (accented chars, CJK) -- use `keyboard.type()` instead.

---

### E-3 -- Workers launch simultaneously; should stagger

**Location:** `automation.py:143-150`

All workers launch simultaneously -- staggering startup (e.g. 2-5s apart) looks more natural and avoids proxy hammering.

---

### E-4 -- No validation that min_time <= max_time

**Location:** `config_window.py`

No validation that `min_time <= max_time` on any spinbox -- setting `min > max` causes a `ValueError` crash in workers.

---

### E-5 -- Synthetic JS MouseEvent blocked by popup prevention

**Location:** `adsense.py:298-313`

Synthetic JS `MouseEvent` with `button: 1` is blocked by browsers as popup prevention -- falsely returns `True`, masking failed ad clicks.
